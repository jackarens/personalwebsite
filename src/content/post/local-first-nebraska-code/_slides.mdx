## Local-first Software

<ul class="p-12">
	<Fragment as="li">
		The **local copy** of your data is effectively complete<sup>*</sup>
	</Fragment>
	<Fragment as="li">
		Server sync happens in the background, off the critical path
	</Fragment>
	<Fragment as="li">Reads and writes feel instant, online or offline</Fragment>
</ul>

<Fragment as="div">
	<Callout type="note" title="Coined by Ink & Switch, 2019">
		"Local-first software: you own your data, in spite of the cloud" —
		Kleppmann, Wiggins, van Hardenberg, McGranaghan
	</Callout>
</Fragment>

<Fragment as="div" class="mt-4 text-sm">
	<span class="opacity-60">* the server still holds the authoritative copy - more on that later</span>
</Fragment>
</Slide>

<Slide>
## A Few of the "7 Ideals"

<ol class="p-12">
	<Fragment as="li">**Fast**: no spinners for your own data</Fragment>
	<Fragment as="li">**Offline-capable**: fully, not degraded</Fragment>
	<Fragment as="li">**Multi-device**: your data follows you</Fragment>
	<Fragment as="li">**Collaboration**: multiple people, same data, at the same time</Fragment>
</ol>
</Slide>

<Slide>
## The Spectrum

| Style                     | Writes Land First On   |   Offline    | Example              |
| ------------------------- | ---------------------- | :----------: | -------------------- |
| Traditional CRUD          | Server                 |      ❌      | Most CRUD SaaS       |
| Offline-available (cache) | Server (queued if not) |   Partial    | Many mobile apps     |
| **Local-first**           | **Client**             |      ✅      | Linear, Figma, Layer |
| Local-only                | Client only            | ✅ (no sync) | Obsidian vaults      |

</Slide>

## Push & Pull Endpoints

<div class="two-column items-start!">
  <div>

    ### Pull

    <Fragment as="div">
    ```ts
    function pull(cookie):
      // Get changes since client's last sync point
      changes = getChangesSince(cookie)

      // Build patches from changes
      patches = buildPatches(changes)

      newCookie = getCurrentVersionCookie()

      // Return patches and new sync point
      return { patches, cookie: newCookie }
    ```
    </Fragment>

  </div>
  <div>

    ### Push

    <Fragment as="div">
    ```ts
    function push(clientId, mutations):
        for each mutation in mutations:
            // Skip if already processed (idempotency)
            if alreadyProcessed(clientId, mutation.id):
                continue
            if !hasPermission(clientId, mutation):
                continue

            // Apply mutation to server state
            applyMutation(mutation)
            recordMutationProcessed(clientId, mutation.id)

        return { success: true }
      ```
    </Fragment>

  </div>

## Conflict Resolution

**Our answer: last write wins - with receipts**

<ul class="p-12">
	<Fragment as="li">
		Two clients edit the same field → the later push wins
	</Fragment>
	<Fragment as="li">Every change is logged - full history per element</Fragment>
	<Fragment as="li">
		Users can roll back any change - an undo beats a merge dialog
	</Fragment>
	<Fragment as="li">
		Same log doubles as change tracking (who, what, when)
	</Fragment>
</ul>

<Fragment as="div">
	<Callout type="note" title="Why this works for us">
		Construction data rarely collides on the same field. When it does, "put it back how it was"
		is what users actually want.
	</Callout>
</Fragment>

## Bootstrapping Large Projects

**Problem**: 40k+ element projects = ~150 - 200MB, slow initial load

<ul class="p-12">
	<Fragment as="li">Multiple round trips (1k–10k batches)</Fragment>
	<Fragment as="li">Serialization time on each request</Fragment>
	<Fragment as="li">
		Pull is called repeatedly until it returns no patches (complete)
	</Fragment>
	<Fragment>
		<Callout type="tip" title="Key Insight">
			Sync engines can send _slightly stale_ data initially - the final pull
			catches up.
		</Callout>
	</Fragment>
</ul>

</Slide>

<Slide>

### Solution: Pre-serialized Bundles

<ul class="p-12">
	<Fragment>
		<Callout type="note" title="Prior Art">
			Inspired by Firestore's bundling mechanism for reducing database reads
		</Callout>
	</Fragment>
	<Fragment as="li">Pre-serialize project data to storage bucket</Fragment>
	<Fragment as="li">Client fetches zipped bundle directly</Fragment>
	<Fragment as="li">70–90% of data instantly</Fragment>
	<Fragment as="li">Background sync for latest changes</Fragment>
	<Fragment as="li">Particularly helpful for projects that start out huge</Fragment>
</ul>
</Slide>

## Bundle Loading Code

<div class="two-column items-start!">
  <div>

    ### Client Side

    <Fragment as="div">
    ```ts
    function pull():
        lastCookie = getStoredCookie()

        response = POST /pull { cookie: lastCookie }

        if response.hasBundleUrl:
            // Bundle exists in storage - load full snapshot
            bundleData = downloadFromStorage(response.bundleUrl)
            applyPatches(bundleData.patches)
            storeCookie(bundleData.cookie)

            // Pull again for changes since bundle was created
            pull()
        else:
            // No bundle - apply incremental patches
            applyPatches(response.patches)
            storeCookie(response.cookie)
      ```
    </Fragment>

  </div>
  <div>

    ### Server Side

    <Fragment as="div">
    ```ts
    function handlePull(cookie):
        changesSinceCookie = getChangesSince(cookie)

        if cookie.fromVersion === 0:
            // Fresh client - serve a pre-built bundle instead
            bundleUrl = getLatestBundleUrl()
            return { bundleUrl, cookie: bundleCookie }
        else:
            // Few changes - return incremental patches
            patches = buildPatches(changesSinceCookie)
            return { patches, cookie: currentCookie }
    ```
    </Fragment>

  </div>
</div>

## Filtering & Sorting

**Problem**: Loading huge amounts of data (10,000+ elements) into memory is slow

<ul class="p-12">
	<Fragment as="li">To sort data, you have to load it all into memory</Fragment>
	<Fragment as="li">Servers/databases solve this with indexing</Fragment>
	<Fragment as="li">Replicache has some support for indexing, but not first class</Fragment>
	<Fragment as="li">Filtering & sorting has similar gaps</Fragment>
	<Fragment as="li">Solution: **LokiDB** for in-memory indexing</Fragment>
	<Fragment as="li">Only index data needed for filter/sort</Fragment>
	<Fragment as="li">Watch Replicache for live updates</Fragment>
</ul>
</Slide>

## Update Contention

**Problem**: Backend calls to update data can fail if made in bulk due to contention

<ul class="p-12">
	<Fragment as="li">
		Client doesn't run into this issue - can retry at will/long lived sessions
	</Fragment>
	<Fragment as="li">Local changes can't have contention</Fragment>
</ul>
 

<Fragment as="h3">Solution: Queue + Retry</Fragment>
<ul class="p-12">
	<Fragment as="li">Cloud queue for retrying failed updates</Fragment>
  <Fragment as="li">Failed updates are retried with backoff until they succeed</Fragment>
  <Fragment as="li">Optionally bulk-pull similar entries from queue</Fragment>
</ul>
</Slide>

<Slide>
## Queue + Retry Code

<div class="two-column items-start!">
  <div>

    ### Enqueue

    <Fragment as="div">
    ```ts
    async function bulkUpdateElements(projectId, updates) {
      await queue.enqueue({
        projectId,
        updates,
        retryCount: 0,
        maxRetries: 5,
      });
    }
    ```
    </Fragment>

  </div>
  <div>

    ### Process + Backoff

    <Fragment as="div">
    ```ts
    async function processQueueItem(item) {
      try {
        await applyUpdatesInTransaction(
          item.projectId,
          item.updates
        );
      } catch (error) {
        if (isContentionError(error) &&
            item.retryCount < item.maxRetries) {
          const delay = 2 ** item.retryCount * 1000;
          await queue.enqueue(
            { ...item, retryCount: item.retryCount + 1 },
            delay
          );
        } else {
          await logFailedUpdate(item, error);
        }
      }
    }
    ```
    </Fragment>

  </div>

## Tradeoffs: When *Not* to Reach for This

<ul class="p-12">
	<Fragment as="li">
		Data needs server-enforced authorization on *every* read
	</Fragment>
	<Fragment as="li">Dataset can't reasonably fit on a client device</Fragment>
	<Fragment as="li">
		Heavy real-time collaborative editing — consider a CRDT library instead of
		rolling your own
	</Fragment>
	<Fragment as="li">
		Small team without bandwidth to own extra client + infra complexity
	</Fragment>
</ul>

<Fragment as="div">
	<Callout type="warning" title="No silver bullet">
		You're not removing complexity - you're just making your users' laptops pay for it
	</Callout>
</Fragment>
</Slide>

<Slide>
## Questions Worth Asking First

<ul class="p-12">
	<Fragment as="li">Where does data authority live? Client, server, or between?</Fragment>
	<Fragment as="li">What environments? Browser, mobile, fully offline?</Fragment>
	<Fragment as="li">How big does one scope's dataset get?</Fragment>
	<Fragment as="li">Who can see it — and when do you check?</Fragment>
	<Fragment as="li">Two people edit the same record. Now what?</Fragment>
	<Fragment as="li">Eventual consistency, or right *now*?</Fragment>
</ul>
</Slide>

<Slide>
## Lessons to Consider

<ol class="p-12">
	<Fragment as="li">
		Replicache fit because it spoke NoSQL - match the engine to your database, not the hype
	</Fragment>
	<Fragment as="li">Your users' devices are part of your infrastructure now - budget for them</Fragment>
	<Fragment as="li">We waited until a 40k-element project hurt - design your bootstrap story first</Fragment>
	<Fragment as="li">Bulk jobs found cracks the client never hit - expect contention at the edges</Fragment>
</ol>
</Slide>

<Slide>
## If You Remember Three Things

<ol class="p-12">
  <Fragment as="li">Local-first means the data is *already there* - that's the whole pitch</Fragment>
  <Fragment as="li">You don't remove complexity, you relocate it - and you sign up to own it</Fragment>
  <Fragment as="li">The sync engine was the easy part - bootstrap, indexing, and contention were the real work</Fragment>
</ol>
